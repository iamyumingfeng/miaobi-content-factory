## 二、核心架构设计

### 2.1 多租户数据隔离

系统采用 `owner_operator_id` 字段实现多租户数据隔离：

```
┌─────────────────────────────────────────────────────────┐
│                    超级管理员 (Super Admin)              │
│  • 拥有系统全部权限                                      │
│  • 可管理所有创作管理员和创作者                            │
│  • 可配置全局模型和系统参数                               │
│  • owner_operator_id = NULL                             │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                 创作管理员 (Operator)                      │
│  • 管理自己的模板、素材、内容生成、分发                    │
│  • 管理自己绑定的创作者                                   │
│  • 使用 owner_operator_id 隔离数据                       │
│  • owner_operator_id = NULL                             │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    创作者 (Sub User)                      │
│  • 仅可查看被分发的内容                                   │
│  • 复制/下载素材，确认发布                               │
│  • 属于某一创作管理员                                    │
│  • owner_operator_id 必填                               │
└─────────────────────────────────────────────────────────┘
```

**数据隔离实现**：
- 所有资源表（template、material、generation_task、generation_item）包含 `owner_operator_id` 字段
- 创作管理员只能访问自己创建的资源
- 创作者只能访问所属创作管理员的资源
- 超级管理员可访问所有数据（通过 `owner_operator_id IS NULL` 或全局查询）

**模型平台配置**：

| 平台 | 平台ID | 模型类型 | 认证方式 |
|------|---------|---------|---------|
| 阿里云百炼 | bailian | llm, image | API Key |
| 火山引擎 | volcano | llm, image | API Key |
| 即梦 | jimeng | image, video | AccessKey + SecretKey |
| 可灵 | kling | image, video | AccessKey + SecretKey |
| AutoGLM | autoglm | llm, image | API Key |
| 月之暗面 | moonshot | llm | API Key |
| DeepSeek | deepseek | llm | API Key |

**认证方式说明**：
- **标准 API Key**：6个平台使用单一 API Key 认证
- **AccessKey + SecretKey**：即梦、可灵使用双密钥认证方式

### 2.5 创意种子库架构

创意种子库用于强制差异化生成，避免AI内容趋同：

```
┌─────────────────────────────────────────────────────────┐
│                    创意种子库架构                         │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  开头模式     │  │  情感基调     │  │  结尾模式     │  │
│  │  (opening)   │  │  (emotion)   │  │  (ending)    │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│        │                  │                  │          │
│        └──────────────────┼──────────────────┘          │
│                           │                              │
│                    ┌──────▼──────┐                       │
│                    │  种子组合    │                       │
│                    │  (随机选择)  │                       │
│                    └──────┬──────┘                       │
│                           │                              │
│                    ┌──────▼──────┐                       │
│                    │  提示词构建  │                       │
│                    │  (整合种子)  │                       │
│                    └──────┬──────┘                       │
│                           │                              │
│                    ┌──────▼──────┐                       │
│                    │  内容生成    │                       │
│                    │  (LLM调用)  │                       │
│                    └──────┬──────┘                       │
│                           │                              │
│                    ┌──────▼──────┐                       │
│                    │  去重检测    │                       │
│                    │  (相似度)    │                       │
│                    └──────┬──────┘                       │
│                           │                              │
│              ┌────────────┼────────────┐                 │
│              │            │            │                 │
│        ┌─────▼─────┐ ┌────▼────┐ ┌─────▼─────┐          │
│        │  通过      │ │  失败   │ │  重试     │          │
│        │  保存      │ │  记录   │ │  切换种子 │          │
│        └───────────┘ └─────────┘ └───────────┘          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**种子组合机制**：
- 系统预置种子：7种开头 + 7种情感 + 6种结尾
- 理论组合数：7 × 7 × 6 = 294 种
- 每次生成使用不同组合，确保内容差异化

**优化效果**：
- LLM调用次数：从7-9+次优化到1-5次
- 重试策略：去重失败时自动切换种子组合
- 内容质量：强制差异化框架，避免AI趋同性

## 四、数据库架构

### 4.1 表设计原则

1. **三表分离设计**: super_admin / operator / sub_user 三张独立表，通过 `UserBase` 抽象基类确保结构一致
2. **多租户隔离**: 所有业务表包含 `owner_operator_id` 字段，实现数据隔离
3. **读写分离预留**: 支持后续扩展读写分离架构
4. **分区策略**: generation_item 按周分区，content_embedding 按月分区，支持海量数据

### 4.3 四级层次结构

**模板四级层次**：
```
TemplatePlatform (模板平台)
    └── TemplateCategory (模板分类)
        └── TemplateTag (模板标签)
            └── Template (模板)
                └── TemplateAttachment (模板附件)
```

**素材四级层次**：
```
MaterialPlatform (素材平台)
    └── MaterialCategory (素材分类)
        └── MaterialTag (素材标签)
            └── Material (素材)
                ├── MaterialAttachment (素材附件)
                └── MaterialFavorite (素材收藏)
```

**设计说明**：
- 每个创作管理员可独立创建自己的平台、分类、标签体系
- 通过 `owner_operator_id` 实现完全数据隔离
- 支持灵活的标签关联（多对多关系）

## 七、安全架构

### 7.1 认证流程（JWT双Token机制）

系统采用 **JWT双Token机制**，包含 `access_token` 和 `refresh_token`：

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  用户   │────▶│  API    │────▶│   DB    │
│  登录   │     │  验证   │     │  查询   │
└─────────┘     └─────────┘     └─────────┘
     │               │               │
     │  1.用户名密码 │ 2.验证凭证    │
     │──────────────▶│─────────────▶│
     │               │               │
     │  3.返回双Token│ 4.返回用户信息 │
     │◀──────────────│◀─────────────│
     │               │               │
     │  access_token (24h)          │
     │  refresh_token (7天)         │
```

**Token刷新流程**：
```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  前端   │────▶│  API    │────▶│   DB    │
│ Token过期│    │  刷新   │     │验证Token│
└─────────┘     └─────────┘     └─────────┘
     │               │               │
     │ 1.refresh_token              │
     │──────────────▶│               │
     │               │ 2.验证有效性  │
     │               │─────────────▶│
     │               │               │
     │ 3.新access_token             │
     │◀──────────────│◀─────────────│
```

### 7.2 JWT Token 结构

**Access Token**：
```json
{
  "sub": "123",              // 用户数据库ID
  "user_type": "operator",   // 用户类型：super_admin/operator/sub_user
  "userid": "O_abc123",      // 用户登录ID
  "exp": 1711680000,         // 过期时间（24小时）
  "iat": 1711676400          // 签发时间
}
```

**Refresh Token**：
```json
{
  "sub": "123",              // 用户数据库ID
  "user_type": "operator",   // 用户类型
  "userid": "O_abc123",      // 用户登录ID
  "jti": "unique_token_id",  // Token唯一标识
  "type": "refresh",         // Token类型标识
  "exp": 1712275200,         // 过期时间（7天）
  "iat": 1711676400          // 签发时间
}
```

**Token存储**：
- `access_token`：前端内存或 localStorage 存储
- `refresh_token`：
  - 前端：localStorage 存储
  - 后端：`refresh_token` 表存储（包含 jti、expires_at）

### 7.3 安全措施

- **HTTPS 全链路加密**
- **JWT Token 时效控制**：
  - access_token：24小时（可配置）
  - refresh_token：7天
- **登录失败锁定**：连续5次失败锁定15分钟
- **数据库 SQL 注入防护**：SQLAlchemy ORM 参数化查询
- **XSS 防护**：前端输入过滤，后端输出转义
- **CORS 跨域控制**：白名单域名验证
- **API 限流**：基于 Redis 的令牌桶算法
- **密码哈希**：bcrypt（rounds=12）
