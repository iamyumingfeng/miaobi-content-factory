## 报告安全漏洞

如果你发现安全漏洞，**请不要通过 GitHub Issues 公开报告**。

### 如何报告

请通过以下方式私密报告安全漏洞:

1. **邮件报告**: 发送详细信息到 **[安全邮箱]**
2. **GitHub Security**: 使用 [GitHub Security Advisories](../../security/advisories) 功能

**注意**: 请勿通过 GitHub Issues 公开报告安全漏洞。

### 报告内容

请包含以下信息：

- 漏洞类型（例如：SQL 注入、XSS、认证绕过等）
- 受影响的功能和文件路径
- 复现步骤
- 概念验证或漏洞利用代码（如有）
- 漏洞的影响范围
- 可能的修复建议（可选）

### 响应时间

- **确认收到**：24小时内
- **初步评估**：3个工作日内
- **修复时间**：根据严重程度，通常在7-14天内

### 漏洞严重程度

| 等级 | 描述 | 响应时间 |
|------|------|----------|
| 严重 | 远程代码执行、数据泄露等 | 24小时内响应，48小时内修复 |
| 高危 | 认证绕过、权限提升等 | 3天内响应，7天内修复 |
| 中危 | XSS、CSRF等 | 7天内响应，14天内修复 |
| 低危 | 信息泄露、轻微逻辑问题 | 14天内响应，30天内修复 |

## 安全最佳实践

### 部署安全

1. **环境变量**：
   - 不要在生产环境使用 `.env.example` 中的示例值
   - 定期更换 `SECRET_KEY`
   - 使用强密码和复杂密钥

2. **数据库安全**：
   - 不要使用默认的数据库密码
   - 限制数据库访问权限
   - 定期备份数据

3. **网络安全**：
   - 使用 HTTPS
   - 配置防火墙规则
   - 限制 API 访问频率

4. **认证安全**：
   - 启用登录失败锁定
   - 定期更换 JWT 密钥
   - 不要共享账号

### 代码安全

- 不要提交敏感信息到 Git
- 使用参数化查询防止 SQL 注入
- 验证和过滤用户输入
- 使用 HTTPS 进行 API 调用
